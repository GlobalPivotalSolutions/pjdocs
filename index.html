What problem does the Tenda project wiki address?

Software development projects require a lot of "paperwork" in the form of requirements documents, design documents, test plans, schedules, checklists, release notes, etc. Trying to keep track of all the documents can be a royal pain, so we are going to use a sort of wiki to keep track of all the work that we do. Using this project wiki will ensure that this project is properly managed.

We are operating to extremely tight timelines and we cannot afford a mis-step. So this technical rollout plan outlines the activities and engineering decisions that have been taken, the methodology we are using, and the timelines and deliverables from each team member. We will try to give a rationale for each proposed decision and some background on why we have chosen a specific path. Folk: Documentation is important - If we reach a verbal decision the affected functional lead should update this wiki with the relevant notes and documents. Don't tell us stuff from textbooks, we can read that as well as you - give a reference or citation, don't quote or copy paste

What is the goal of this documentation effort?
We want a:
  • Reproducible process for creating reliable software
  • Visibility into the development process
  • Best Practice
  • We want to limit the DevOps challenges in deployment
  • Maintainability and rapid evolution to satisfy market requirements
Key features that define the Tenda Document Set (DocSet)
  • We ONLY use HTML and text based formats so that we are not continuously changing tools from coding to dcumentation. That is why we use LaTeX and HTML for documentation. if you use MS word or some such - Fie on you, Most of us won't read it, our computers are busy enough as it is without firing up yet another application
  • High-quality Documentation. (Be as thorough as you can be) If you use or do something new or innovative, please point it out here so that the rest of the team can see what it is
  • We use standardized documents for many common software engineering processes. Including:
    • Project proposal
    • Project plan (use TaskJuggler)
    • Use cases (use Yaoqiang BPMN Editor )
    • QA plan
    • Design templates
    • System test case
    • Release checklists

Scope

What is the scope of this project?

We are realizing not just software, but a SYSTEM. It involves People (Civil Servants, Users, and Us), Processes (which we must try to make as easy as possible) and Technology (what we are building). We are building a system to surface government procurement to the general population, to facilitate broad participation in Government procurement, and hopefully build a reliable, trustworthy database of products, prices, actors and activities that we can use to study trends and general tendencies in our country.

We are building an Open Source System - Anybody will be able to see how we have built this and hopefully use our code for other purposes. (So: No flippant or personal comments in the code)

What are the high-level assumptions or ground rules for the project?
  • We will use only open source technologies and tools
  • We are developing an open source product
  • The reusltant products will be freely available to anybody to use
  • Security of the data is important
  • Don't re-invent the wheel
  • When the design is finished we will have a check-in every other day
  • The Project plan says you have Sunday's off (but you should know better)
  • We have very little slack in our project plan (2 weeks) - Marketing and Meeting will take up most of it, not development (You have been warned)
