The setting and history behind Tenda

The GoK makes thousands of purchases every week. By policy it has been established that 30% of all procurement is reserved for Youth, Women and Persons with Disabilities. The effectiveness of this provision has been mitigated by implementational difficulties.

One of the principal challenges to the success of the AGPO inititive (as it is called) is that the Youth, Women and Persons with disabilities do NOT know about these opportunities when they arise. These opportunities (when they are made public) are published in newspapers. In Kenya newspapers are of very limited circulation. Kenya has 4 Newspapers with national coverage, with a total circulation fo 250,000. I a population just north of 40M people, it is a trite to say that nobody reads newspapers! In 2014, Four million Kenyans, roughly 10 percent of the population, engaged daily in social media activities with 570 million tweets expected by the end of 2014 and more than 50 thousand new twitter accounts predicted at the same period of time. This is the result of the research by The Digital Rand on “The State of Social Media in Kenya” which was published in 2014. According to statistics on Social Bakers, majority of these users age between 18-24 and 25-34 years with 64% of them being male and 36% female.

Tenda will provide a means for this online/non-newspaper reading population to get information in tenders and government procurement on their smartphones.

What is the problem to be addressed?
There are hundreds of government procuring entities, each with its own website and newspaper advertisements. (Each university, college, parastatal, school advertizes independently). If a youthful or female entrepreneur wanted to keep track of these hundreds of website to find out if a deal they were interested was available, it would be a onerous, timeconsuming daily activity. There is a need for a faster way for users to find information of interest, immediately it becomes avalable.
Some current approaches to this problem
Users can tell each other about tenders, but that is not scalable because it depends on manual steps by people who may not be motivated or honest in their evaluations. There are some tender directory web sites, but they are not automated so they are always out of date and do not rate the quality of the tenders.

The treasury tender website list tenders that have expired.

  • Treasury Tender Website
  • PPOA Tender Listing (links right back to the useless Treasury list!)
Why this problem worth solving or worth solving better
A better tender directory service would be valuable because it could greatly increase availability of business, entrepreneur satisfaction satisfaction and allow them to spend more time responding to tenders and executing work and less time searching.

The lack of competition, opaque processes, and back room deals is the very source of public procurement corruption. We can help eradicate corruption by bringing these opportrunities to light to a larger community of entrepreneurs, speccifically the under-represented.

How Tenda will be better than previous approaches
We add innovative new features. Our tender directory will be larger an more detailed. We will offer built-in tools for managing responses, searching for sepecific opportunities, get real time alerts when opportunities emerge and organizing consortia to repsond to Tenders nights or tournaments.
Our system will have substantially different functionality, and it will have much better maintainability, scalability, and security.

Goal

The goal of this project
This project will produce a real-time engine for tender alerts that allows users to quickly find, evaluate,respond and track tenders as they emerge. We will develop a analytical engine to track likelyhood of success, and permit users to form consortia with like-minded people to respond to tenders.
The defining features and benefits of this product set
  • Reusable website engine with functionality for searching for tenders, creating responses, editing, deleting, searching, categorizing, browsing, rating, and commenting on Tenders. This automates all tender response operations and ensures that users will always find information that is automatically up-to-date.
  • The tender alert engine will have a highly configurable appearance that allows it to be used by visually impaired persons, people with limited articulation, and configurable look and feel. This allows the reusable website engine to have a high-quality appearance that is just as appropriate for the target audience.
  • The engine will be secure and only allow users with the proper permissions to respond, edit, delete, or join a consortium. We will use the power of social media at the core of our design.

Scope

We want to focus on the mobile and web applications, and the features of that application that help build a great user experience.

See the context diagram.

  • Work with common servers and browsers that we are already familiar with.
  • Allow easy customizations of fonts and colors, with the same basic layout.
  • Security to greatly discourage abuse
  • The application content discusses tenders, but we do not want to integrate directly with government software - we do not want to be the vector for an attack on public infrastructure. We want to extract data safely not insert or inject data into these systems.
In Scope Out of Scope
Building a web application for use with standard web servers and application servers Building a new web server or application server
Working the most popular browsers (IE6, NN7/Mozilla) Working with uncommon or outdated browsers
Security in the form of user accounts, passwords, and permissions Finding or patching security holes in existing software components.
Open source look-and-feel components (bootstrap/jQuery/Material/Polymer) and instructions for customization A library of look-and-feel options.
Database and server load and data volume that can be handled by one cluster server. Managing a cluster of virtual private servers.
Keeping track of which users have responded to which tenders Tracking offline tender response activity

Main risks of this project
  1. There is a potential conflict between the goals of a high-quality appearance and one that is completely customizable. We can only succeed if users find the web site useful and appealing. We already have a design in mind that will address this risk and we will review it with a UX Expert
  2. There are significant technical difficulties in building a web site and web application. This will be a risk because one person on our team has much experience with the relevant tools and technologies. Although the others will learn, we will certainly make some mistakes and suboptimal choices. We will address this risk by scoping the project such that we have enough time to train and to review the design and implementation.
  3. The schedule for this project is very short. We will manage this by planning a conservatively scoped functional core and series of functional enhancements that can be individually slipped to later releases if needed.
When Tenda succeeds (as it must)
If we accomplish the elements of our plan, our tender directory engine will replace the existing directory websites and generate traffic which will result in increased citizen participation in government procurement. Our ability to overcome the challenges above will determine time to market, the speed of adoption, the amount of web traffic, and thus the generated revenue over time.
